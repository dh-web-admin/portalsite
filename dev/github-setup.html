Git Configuration & Version Control for PortalSite

Setup Version Control ‚Ä¢ Last Updated: November 6, 2025

Overview

This guide covers Git and GitHub setup for the PortalSite project, including initial configuration, daily workflow commands, authentication options, and best practices for version control.

Initial Setup (One-Time)

Step 1: Configure Git

# Configure Git with your info
git config --global user.name "Your Name"
git config --global user.email "your.email@example.com"

Step 2: Initialize Repository

# Initialize repository
git init

# Add all files
git add .

Step 3: First Commit

# First commit
git commit -m "Initial commit: Employee portal with user management"

Step 4: Connect to GitHub

# Connect to GitHub:
git remote add origin https://github.com/dh-web-admin/portalsite.git

# Rename branch to main
git branch -M main

# Push to GitHub
git push -u origin main

Daily Workflow

Check Status

# Check status of files
git status

Shows which files have been modified, added, or deleted.

Stage Changes

# Add changed files
git add .

# Or add specific files
git add path/to/file.php

Stages files for commit.

Commit Changes

# Commit with message
git commit -m "Description of what you changed"

Records your changes with a descriptive message.

Push to GitHub

# Push to GitHub
git push

Uploads your commits to the remote repository.

Pull Latest Changes

# Pull latest changes
git pull

Downloads and merges changes from the remote repository.

Useful Commands

View History

# See commit history
git log

# See compact history
git log --oneline

# See history with graph
git log --graph --oneline --all

View Changes

# See what changed (unstaged)
git diff

# See what changed (staged)
git diff --staged

Branch Management

# Create new branch
git checkout -b feature-name

# Switch branches
git checkout main

# List branches
git branch

# Delete a branch
git branch -d feature-name

Undo Changes

# Undo local changes (before commit)
git checkout -- filename.php

# Unstage file
git reset HEAD filename.php

# Undo last commit (keep changes)
git reset --soft HEAD~1

If You Make a Mistake

Revert Last Commit (Safe)

# Creates new commit that undoes the last commit
git revert HEAD
git push

‚úÖ Safe: Preserves history

Reset to Previous Commit (Destructive)

# Go back to previous commit (destructive)
git reset --hard HEAD~1
git push --force

‚ö†Ô∏è Destructive: Rewrites history

Discard All Local Changes

# Discard all local changes
git reset --hard origin/main

‚ö†Ô∏è Cannot be undone

GitHub Authentication

Option 1: Personal Access Token (Recommended)

  1. Go to GitHub.com ‚Üí Settings ‚Üí Developer Settings
  2. Click Personal Access Tokens ‚Üí Tokens (classic)
  3. Click Generate New Token
  4. Select scopes: repo (full control)
  5. Copy token and save it securely
  6. Use token as password when pushing

Option 2: GitHub CLI (Easier)

# Install GitHub CLI
winget install GitHub.cli

# Authenticate
gh auth login

# Follow prompts to authenticate via browser

Option 3: SSH Keys

# Generate SSH key
ssh-keygen -t ed25519 -C "your.email@example.com"

# Copy public key
cat ~/.ssh/id_ed25519.pub

Add to GitHub: Settings ‚Üí SSH and GPG keys ‚Üí New SSH key

Common Issues & Solutions

‚ùå Issue: "git is not recognized"

Solution: Restart terminal/VS Code after installing Git

‚ùå Issue: Authentication failed

Solution: Use Personal Access Token instead of password

‚ùå Issue: "fatal: not a git repository"

Solution: Make sure you ran git init first

‚ùå Issue: Conflicts when pulling

Solution: 
git stash          # Save your changes
git pull           # Get latest
git stash pop      # Reapply your changes

‚ùå Issue: Pushed sensitive data (config.php)

Solution: 
# Remove from Git history
git filter-branch --force --index-filter \
  "git rm --cached --ignore-unmatch config/config.php" \
  --prune-empty --tag-name-filter cat -- --all

# Force push
git push origin --force --all

‚ö†Ô∏è Change your database passwords immediately!

Best Practices

Commit Messages

‚úÖ Good Examples

  • "Add password reset functionality to user list"
  • "Fix login redirect bug for non-admin users"
  • "Update dashboard tiles to be responsive"
  • "Refactor database connection to use prepared statements"

‚ùå Bad Examples

  • "updates"
  • "fixed stuff"
  • "asdf"
  • "changes"

When to Commit

  • ‚úÖ After completing a feature
  • ‚úÖ After fixing a bug
  • ‚úÖ Before trying something experimental
  • ‚úÖ At the end of the day
  • ‚úÖ Before switching to different work
  • ‚ùå Don't commit broken code to main
  • ‚ùå Don't commit commented-out code

What to Commit

‚úÖ DO Commit

  • Source code (.php, .css, .js, .html)
  • Documentation (.md, .html files)
  • Configuration templates (.example files)
  • Database schema files (.sql)
  • Asset files (images, fonts)

‚ùå DON'T Commit

  • Actual config files with passwords
  • Database files (.db, .sqlite)
  • Log files (.log)
  • Temporary files (.tmp, .bak)
  • IDE settings (unless shared)
  • node_modules/ or vendor/

Branches Strategy

Main Branch

  • Production-ready code
  • Always stable
  • Protected from direct commits
  • Only merge tested code

Development Workflow

1. Start New Feature

3. Test Feature

Test thoroughly before merging

4. Merge to Main

Deployment with Git

Pull to Server

# On your production server
cd /var/www/html/PortalSite

# Restart services if needed
sudo systemctl restart apache2

Rollback Production

# If something breaks in production
git revert HEAD
git push

# On server
git pull origin main
